## 0. 背景：为什么要这个 Worker

### 0.1 um-api 是什么，它负责解密吗？
**不负责。** 解密 100% 在你的浏览器本地（前端 Web Worker 里的 `src/decrypt/*` 纯 JS 算法）完成，**不联网、不调任何远端服务**。断网也能解。

`um-api.ixarea.com` 原本只做两件事（都是"锦上添花"）：
1. **在线封面** —— 给文件本身没内嵌封面的 QQ 曲子补一张专辑图；
2. **在线元数据** —— 歌曲流派/简介/语言等扩展字段。

而且**只有 QQ 系**（`qmc.ts` / `qmc_oj.ts` / `musicex.ts`）调它，网易 NCM、酷狗 KGG、咪咕等其它平台从头到尾不碰它。

### 0.2 um-api 现在的状态
2026-07-23 实测返回 **HTTP 522（Cloudflare 源站超时）** —— 上游已下线/无人维护。
- 解密完全不受影响（前端已 try/catch 降级，文件照常产出）；
- 仅"无内嵌封面的 QQ 文件没封面"、带 id 的文件少点扩展字段。

### 0.3 本 Worker 替代什么
替代 um-api 的**封面**部分。**不碰解密**。当前只实现封面，元数据端点故意留空（未来补，见 §9）。

当前**示例**线上地址（本项目已部署，仅作参考）：
```
https://<你的-worker子域>.workers.dev
```
> 示例值：`unmusic-cover-proxy.xianshenghu363.workers.dev`（本项目实际子域）。复刻时这里会不同，以你部署后 Cloudflare 给的地址为准。

## 1. 它做了什么（端点 + 数据源）

| 端点 | 作用 | 数据源 |
|------|------|--------|
| `GET /music/qq-cover?Title=&Artist=&Album=` | 调 QQ 匿名搜索接口，搜出专辑封面 ID，返回 `{"Id":"...","Type":1}` | QQ `client_search_cp`（带 `Referer: https://y.qq.com` + `User-Agent`） |
| `GET /music/qq-cover/{type}/{pmid}` | 从 QQ 公开 CDN（gtimg）取封面图，并补 `Access-Control-Allow-Origin: *` CORS 头 | `y.gtimg.cn/music/photo_new/T002R300x300M000{pmid}.jpg`（公开标准地址） |
| `GET /music/kugou-cover?Title=&Artist=&Album=` | 调酷狗匿名搜索接口，搜出封面直链，返回 `{"Id":"...","Type":1}`（Id 为 `imge.kugou.com` 直链，已替换尺寸占位符 `{size}`→`480`） | 酷狗 `songsearch.kugou.com/song_search_v2`（带 `Referer: https://www.kugou.com` + `User-Agent`） |
| `GET /music/img?url=<encoded>` | 受限图片代理：仅放行 `kugou.com` 域名，带 `Referer` 取图并补 `Access-Control-Allow-Origin: *` CORS 头，绕过 `imge.kugou.com` 的 Referer 防盗链 | 酷狗图床 `imge.kugou.com/...`（前端拿到上一条的 Id 后，经此代理取真实图片） |
| `GET /meta/qq-music-raw/{id}` | **本期未实现**，直接返回 `{"code":-1}` | 前端收到后自动回退本地元数据，不影响解密 |

- 前端 **QQ 系**（`qm_meta.ts`）调 `/music/qq-cover*`；**酷狗 KGG**（`kgg.ts`）调 `/music/kugou-cover` + `/music/img`。其余平台（NCM、咪咕等）不碰。
- 所有请求都带 CORS 头 `Access-Control-Allow-Origin: *`，所以即使 Worker 和你的网页（pages.dev）不同源，浏览器也能正常取封面（不像之前 um-api 被 CORS 拦）。

## 3. 本地手动部署 / 更新（临时改代码时用）

> 只在"你在本机改了 `src/index.js` 想立刻生效"时才用。日常更新请走 §5 自动部署或 §7 前端更新流程。

前置：本机已装 Node（≥16.17，本项目实测 Node 24 可用）。

```bash
cd source/works/cover-worker
npx wrangler login        # 首次：浏览器弹窗授权 Cloudflare 账号（一次性，之后不用再登）
npx wrangler deploy       # 部署 / 更新（改完 index.js 后重跑这条即可）
```

部署成功终端会输出你的地址：
```
https://unmusic-cover-proxy.<你的子域>.workers.dev
```

> ⚠️ **只有改了本目录的代码才需要 `wrangler deploy`**；改前端业务代码不需要碰 Worker。

## 5. 托管 GitHub + 自动部署（Cloudflare Connect to Git，极简推荐）

**不需要你手写任何配置文件，也不需要自己配 Token / Secret。** 认证由 Cloudflare 的 GitHub 集成自动处理。

### 5.1 在 Cloudflare 控制台连接仓库（逐步 + 表单字段详解）

1. 登录 Cloudflare 控制台 → 左侧 **Workers & Pages** → 点 **Create** → **选「Connect to Git」**（注意：不是创建空白 Worker）。
2. 授权连接你的 GitHub 账号，在仓库列表里选中 **你的仓库**（示例：`unmusic`）。
3. 进入配置表单，**逐项这样填**：

   | 表单字段 | 填什么（示例，需替换成你自己的） | 说明 |
   |---------|--------|------|
   | Git 仓库 | `你的用户名/仓库名`<br>（示例：`HTryone/unmusic`） | 选你 fork 后的仓库；**不要照抄示例**，换成你自己的 |
   | 生产分支 | `main`（或你的主分支名） | 主分支，按你仓库实际命名 |
   | **根目录（Root directory）** | **`works/cover-worker`** | ⚠️ 最关键！必须指向子目录，Cloudflare 才能找到 `wrangler.toml`；留空会找不到配置导致部署失败（若你把 Worker 放在别的目录，填你的实际目录名） |
   | **构建命令（Build command）** | **留空（不填）** | ❌ 不要填 `npm run build`！那是前端（source/ 里的 Vue 项目）用的，`works/cover-worker/` 没有构建步骤 |
   | 部署命令（Deploy command） | `npx wrangler deploy` | 正确，保持 |
   | 版本命令（可选） | `npx wrangler versions upload` | Cloudflare 自动加的，保持默认即可 |
   | **构建监视路径（Build watch path）** | **`works/cover-worker/**`** | ⚠️ 默认是 `*`(监听所有路径)。改成 `works/cover-worker/**`，这样只有 Worker 目录有改动才触发部署，改前端不会误触发（若你改了根目录，这里要同步改） |
   | 非生产分支构建 | 可勾选 | 不碍事，保持默认 |

4. **API 令牌**：选「创建新令牌 / 自动创建」。弹出令牌表单，**这样填**：
   - **令牌名称**：随意，默认 `Workers 构建 - <日期>` 即可；
   - **权限**：Cloudflare 已自动勾好一组，最关键的是 **Account → Workers Scripts → Edit**（部署必需，**别删**），其余保持原样；
   - **账户资源**：选「包含 → 特定账户 → 你的账户」（示例账户名：`xianshenghu363`），确认是你的号；
   - **区域资源**：纯 Worker 选「无 (None)」即可，或「所有区域」都行；
   - **令牌有效期 (TTL)**：建议选 **「永不过期」**，否则到期 Git 自动部署会失效；
   - **客户端 IP 地址过滤**：留空（不限制）。
   点「创建令牌」即可（自动模式不用你复制粘贴）。
5. 确认创建。控制台会显示「现在您可以将提交推送到 Git 存储库来开始第一次构建了」。

### 5.2 触发第一次自动部署

连接完成后，往 `main` 推一次**改动了 `works/cover-worker/` 的提交**即可触发。最简单：给 `works/cover-worker/部署指南.md` 末尾加一行空行再提交：

```bash
cd source
git add works/cover-worker
git commit -m "触发 Worker 自动部署"
git push origin main
```

push 后 Cloudflare 会自动执行 `npx wrangler deploy`，在控制台该项目的**部署日志**里能看到结果。
（注意：因为 5.1 改了监视路径为 `works/cover-worker/**`，**必须这次提交里包含 works/cover-worker/ 的改动**才会触发；只改前端不会触发。）

### 5.3 以后怎么更新 Worker
1. 本地改 `works/cover-worker/src/index.js`；
2. `git add works/cover-worker && git commit && git push origin main`；
3. Cloudflare 自动部署，完事 —— 再也不用手动 `wrangler login/deploy`。

## 7. 日常更新流程（区分前端 vs Worker，别搞混）

很多人会问"以后每次部署都要 wrangler 登录/部署吗？" —— **不需要。** 分清两件事：

### 7.1 前端代码更新（你以后的常规操作，与 Worker 无关）
```
cd source
npm run build              # 出 dist/
把 dist/ 整个发到 pages.dev   # 你原来的发布方式
```
`api.ts` 里那个 Worker 地址已经写死，build 自动带上，你**不用每次改它**。

### 7.2 Worker（封面代理）
它是一次性资产，地址固定。
- **只要你不改 `works/cover-worker/src/index.js`，就永远不用再 `wrangler deploy`**；
- 只有将来扩展它（比如 §9 加元数据端点）时，改完代码 `git push` 即可，Cloudflare 自动重新部署。

一句话：**今天那套"登录、部署 Worker"的折腾只发生在首次接通，不会重复。** 改前端业务代码 = 就和以前一样 build + 发 dist。

## 8. 验证部署是否成功

### 8.1 验证 Worker 两个端点（用 curl 或浏览器）
```bash
# ⚠️ 下面三条命令里的 <你的-worker子域> 换成你自己的地址（示例：unmusic-cover-proxy.xianshenghu363）
# 1. 搜封面：应返回 JSON，含 Id / Type
curl "https://<你的-worker子域>.workers.dev/music/qq-cover?Title=%E4%BF%9D%E7%95%99&Artist=%E9%83%AD%E9%A1%B6&Album=%E9%A3%9E%E8%A1%8C%E5%99%A8%E7%9A%84%E6%89%A7%E8%A1%8C%E5%91%A8%E6%9C%9F"

# 2. 取封面图：应返回 HTTP 200 image/jpeg
curl -o test.jpg "https://<你的-worker子域>.workers.dev/music/qq-cover/2/000MkMni19ClKG"

# 3. 看 CORS 头（关键，证明 pages.dev 能跨域取）
curl -I "https://<你的-worker子域>.workers.dev/music/qq-cover/2/000MkMni19ClKG" | grep -i access-control
# 期望看到： access-control-allow-origin: *

# 4. 酷狗搜封面：应返回 JSON，含 Id(图片直链)/Type
curl "https://<你的-worker子域>.workers.dev/music/kugou-cover?Title=%E5%91%A8%E6%9D%B0%E4%BC%A6&Artist=%E5%91%A8%E6%9D%B0%E4%BC%A6&Album=%E5%8F%B6%E6%83%A0%E7%BE%8E"

# 5. 酷狗图片代理：把上一条拿到的 Id 编码后传 url，应返回 HTTP 200 image/jpeg
#    （url 需 URL-encode，下面用 --data-urlencode 让 curl 自动编码）
curl -o kugou.jpg --get "https://<你的-worker子域>.workers.dev/music/img" --data-urlencode "url=http://imge.kugou.com/stdmusic/480/20230920/20230920142503632013.jpg"
```

### 8.2 验证前端封面恢复
- 本地：`http://localhost:4173/` 打开，**Ctrl+Shift+R 强刷**（清旧 PWA Service Worker 缓存）；
- 拖一个"文件内嵌无封面"的 QQ 曲子进去，看封面是否出现；
- 线上：把 `dist/` 发到 `<你的 pages.dev 站点>` 后同样验证。

## 10. 常见疑问 FAQ（你这一路问过的，逐一答）

**Q1：um-api 负责解密吗？**
不负责。解密全在浏览器本地完成，跟任何远端服务无关。um-api 只补封面/元数据，且只限 QQ 系。

**Q2：旧的后端是不是全都换成我们的了？**
代码层面零残留——`api.ts` 那个常量已完全指向我们的 Worker，没有任何地方还调 um-api。功能上：封面已 100% 由我们 Worker 提供；元数据暂时由前端本地兜底（Worker 故意返回 `{code:-1}`，等 §9 补）。也就是"旧后端已不被调用"，只是元数据还没在我们这边重新提供。

**Q3：以后每次部署都要 wrangler 登录/部署吗？**
不用。首次接通才需要。以后改前端 = `build` + 发 `dist`；只有改了 `works/cover-worker/src/index.js` 才需要重新部署（且已配好 Git 自动部署，push 即生效）。

**Q4：构建完成了怎么用 dist？**
`npm run build` 出的 `source/dist/` 整个目录，按你原来的方式发到 `<你的 pages.dev 站点>` 即可。Worker 地址在 build 时已编进 `dist`。

**Q5：为什么本地/线上有时看不到新效果？**
前端是 PWA，旧 Service Worker 会缓存 JS。验证时 **Ctrl+Shift+R 强制刷新** 或开无痕窗口。

**Q6：`*.workers.dev` 在国内访问慢/不稳？**
这是 Cloudflare 免费子域的已知现象。两个解法：① 给 Worker 绑你自己的域名（CF 免费支持）；② 以后改用 Cloudflare Pages Functions 同源部署（彻底无 CORS）。当前先用 workers.dev 跑通。

**Q7：元数据端点现在返回 {code:-1}，文件会坏吗？**
不会。前端已 try/catch，拿不到在线元数据就回退本地标签，音频照常解出。

**Q8：构建监视路径 `*` 和 `works/cover-worker/**` 有区别吗？**
有。`*` 会监听仓库所有文件改动（包括改前端）都触发 Worker 重部署，浪费；`works/cover-worker/**` 只在 Worker 目录有改动时才部署。已建议改成后者。

## 11. 排错速查

| 现象 | 原因 / 解决 |
|------|------------|
| Cloudflare 部署报"找不到 wrangler.toml" | 根目录没填 `works/cover-worker`，回去改 |
| 部署时 `npm run build` 报找不到脚本 | 构建命令误填了 `npm run build`，清空它 |
| push 后没触发自动部署 | 这次提交没包含 `works/cover-worker/` 的改动（监视路径是 `works/cover-worker/**`），改个 README 再推 |
| 封面仍取不到 | 先用 §8.1 的 curl 直接打 Worker 端点，确认返回 200；QQ 老接口若被腾讯调整则需改 `index.js` |
| 前端旧页面没刷新 | Ctrl+Shift+R 或清 Service Worker（DevTools → Application → Service Workers → Unregister） |
