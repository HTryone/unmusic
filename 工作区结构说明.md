> 本文档说明 `D:/desktop/unmusic/source/` 这个 Git 仓库里每个目录和文件的用途，并解释「构建 / 打包」是怎么回事。

## 一、总览：工作区由两部分组成

```
D:/desktop/unmusic/
├── source/         ① 迁移到 Vue 3 + Vite 的「源代码」（我们开发的地方）
└── .workbuddy/     ② AI 的工作记忆（自动维护，你不用管）
```

一句话区分：

- **`source/` = 我们自己的源代码**，浏览器看不懂，必须先「构建」才能变成可运行的网站。
- **`source/dist/` = `source/` 构建后的成品**，是可直接托管的「可部署网站」。

## 三、`source/dist/` —— 构建产物（构建后才出现）

执行 `npm run build` 后生成。它**是一个可直接托管的静态网站**。

```
dist/
├── index.html              入口页（已注入打包后的脚本引用）
├── assets/                 编译压缩后的 JS / CSS（带内容哈希文件名）
│   ├── index-xxxx.js       主程序
│   ├── element-xxxx.js     Element Plus 拆分块
│   ├── qmcwasm-xxxx.js     解密 WASM 拆分块
│   ├── crypto-xxxx.js      crypto-js 拆分块
│   ├── metadata-xxxx.js    music-metadata 拆分块
│   └── jimp-xxxx.js        封面处理拆分块
├── img/  favicon.ico  loader.js   从 public/ 原样拷贝
├── manifest.webmanifest    PWA 清单（vite-plugin-pwa 生成）
├── sw.js  registerSW.js  workbox-*.js   Service Worker（离线安装）
```

> 注意：`dist/` 已被 `.gitignore` 排除，**不进 git**。它是「随时可重新生成的产物」，不是源码。要部署时重新 `npm run build` 一份即可。

## 四、重点：构建（打包）到底是怎么回事？

### 4.1 为什么需要「构建」？

浏览器只认三样东西：**HTML、CSS、JavaScript**。但 `source/src/` 里写的是：
- `.vue` 单文件组件（HTML+JS+CSS 混在一起，浏览器不懂）
- TypeScript（类型语法浏览器不懂）
- SCSS（嵌套语法浏览器不懂）
- 几十个互相 `import` 的模块文件

所以必须有一个「翻译 + 打包」的过程，把这一大堆源文件变成浏览器能直接跑的少量 `.js`/`.css`/`.html` —— 这个过程就是**构建（build）**，也叫打包（bundle）。工具在这里是 **Vite 5**。

### 4.2 `npm run build` 到底做了什么？

1. **编译**：`.vue` → JS 渲染函数；TypeScript → JS；SCSS → CSS。
2. **打包（bundle）**：把几百个互相引用的模块，合并成少数几个文件，消除 `import` 关系（浏览器不用再做几百次网络请求）。
3. **压缩（minify）**：删空格、缩短变量名，体积大幅缩小。
4. **代码分割（code splitting）**：按 `manualChunks` 拆成 `element` / `qmcwasm` / `crypto` / `metadata` / `jimp` 等独立块，**用到才加载**（比如你拖入 ncm 才加载 jimp）。
5. **内容哈希命名**：文件名变成 `index-a1b2c3.js` 这种。内容一变哈希就变 → 浏览器能放心长期缓存，更新时自动失效。
6. **WASM 处理**：解密用的 `.wasm`（qmc/kgm）被内联或作为资源产出。
7. **Polyfill**：用 `vite-plugin-node-polyfills` 给浏览器补上 `Buffer`/`process` 等 Node 才有的全局（jimp 等库需要）。
8. **PWA**：`vite-plugin-pwa` 自动生成 `sw.js`、`manifest.webmanifest`、`registerSW.js`，让网站可「安装到桌面 / 离线使用」。

构建命令本身（`package.json` 里）：
```bash
npm run build      # = vite build，产出 dist/
npm run dev        # = vite，开发服务器（:5173，热更新，不用构建）
npm run preview    # = vite preview，本地预览构建后的 dist/
npm run type-check # = vue-tsc，只做类型检查不产出
```

### 4.3 `dist/` 是怎样的、和源码有什么关系？

`dist/` 是 `source/` 这套源码经过「构建」后吐出的**纯静态成品**，和 `source/src/` 里的源文件有本质区别：

| 特征 | `source/src/`（源码） | `dist/`（构建产物） |
|------|------|------|
| 内容 | `.vue` / `.ts` / `.scss`，几十~几百个模块 | 少量 `.js` / `.css` / `.html`，浏览器直接可跑 |
| 文件名 | `Home.vue`、`decrypt/ncm.ts` 这种可读名 | `index-a1b2c3.js` 这种带内容哈希的名 |
| 是否可调试 | 是（有类型、有原始结构） | 否（已压缩、变量名被缩短） |
| 是否进 git | 是（源码要版本管理） | 否（`.gitignore` 排除，随时 `npm run build` 重建） |
| 包含什么 | 组件、解密逻辑、配置 | 编译后代码 + PWA 文件（`sw.js` / `manifest.webmanifest` / `workbox-*.js`） |

**关键点**：`dist/` 里**不含任何源码**，它是源码「编译 + 打包 + 压缩 + 哈希命名 + PWA 注入」的结果。你改功能只动 `source/src/`；改完跑 `npm run build`，`dist/` 就会被重新生成。所以它永远以 `source/` 为准——**不要手改 `dist/` 里的文件**，改了也会被下次构建覆盖。

## 五、日常怎么用（速查）

### 开发 / 改代码时
```bash
cd D:/desktop/unmusic/source
npm install          # 首次需要，安装依赖（已装可跳过）
npm run dev          # 起开发服务器，浏览器开 http://localhost:5173（Vite 默认端口）
```
改 `src/` 里的文件会**热更新**，不用重新构建。

### 想生成一个可部署的网站
```bash
cd D:/desktop/unmusic/source
npm run build        # 产出 dist/
# 然后把 dist/ 整个文件夹拷到任意静态服务器（Nginx / GitHub Pages / Cloudflare Pages）
# 或双击仓库里的 start-dist.bat，用 python 一键起 http://localhost:8787 预览 dist/
# 发行：把整个 dist/ 压缩成 zip 分发，解压后双击其中的 start-dist.bat 即可（详见下节）
```

### 只想用现成的、不想碰代码

**在开发仓库里**：双击 `source/start-dist.bat`（它在 `source/` 下，会进入同级 `dist/` 起服务），浏览器开 `http://localhost:8787` 即可。

**发行给别人（.zip 压缩包）**：
1. `npm run build` 后，`source/dist/` 里**已自带 `start-dist.bat`**（构建时由 `public/` 自动拷入，无需手动复制）。
2. 把整个 `dist/` 文件夹压缩成 zip 分发。
3. 对方解压后，`index.html` 与 `start-dist.bat` **在同一目录**，双击 bat 即可——脚本检测到当前目录没有 `dist/` 子目录，就以自身所在目录为准起服务。
4. 解压位置**随意**（桌面、任意盘均可），不依赖 `source/`，也**不需要 Python 之外的任何依赖**（前提是装了 Python 且 `python` 在 PATH 中）。

---

## 六、小结

- **`source/` = 源代码**，**`source/dist/` = 构建后的成品**。
- 你日常开发和改功能在 `source/src/`；`npm run build` 把源码「翻译 + 打包」成 `dist/`；`dist/` 是一个可直接托管的网站，更新、更现代。
- `dist/` 是纯构建产物，不进 git，随时可重新生成。
